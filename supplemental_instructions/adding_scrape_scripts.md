# Adding a utility scrape script to the backend

Each utility site gets one JavaScript file. The backend loads it at startup and
serves it to the app on demand, matched to a linked account by `site_id`.

Scripts live in `application-plane/backend/ap-utility/scrape_scripts/`.

Scripts are generated by the login-flow-recorder, not hand-written. The recorder
captures the login steps and emits a file with a `NAV_STEPS` array, an
`EXTRACTION` function, and the step-pointer machinery below. The rest of this
doc describes what the generated file contains and the contract it satisfies.

## Steps

1. Name the file `<site_id>.js`. The base name is the `site_id`. It must match a
   `site_id` in the app's provider catalog (see "Adding a site to the app"
   below); the app selects a site by that id and requests its script by that id.
2. Put a `// @url <login-url>` header on its own line. Files without this header
   are skipped at load (`ap-server/src/config.ts`). This header is the only
   place the login URL lives; it is not stored on the device or in the linked
   account record.
3. The file defines a global `run_scrape(creds)` function. It runs inside the
   off-screen WebView, is re-injected on every page load, and must return (or
   resolve a promise to) `{ bills, usage }`.
4. Restart the backend. The registry is built once at startup; new or edited
   files are not picked up until restart.

## Adding a site to the app

The set of selectable providers is a fixed catalog, `PROVIDER_CATALOG` in
`application-plane/mobile/resident-mobile/src/m-res-portal/types.ts`. Each entry
is `{ site_id, provider }`: `site_id` is the id (and the backend script file base
name); `provider` is the display name shown in the add-account dropdown.

To offer a new site:

1. Add the scrape script `<site_id>.js` under `scrape_scripts/` (steps above).
2. Add a matching `{ site_id, provider }` entry to `PROVIDER_CATALOG`.

The `site_id` in both places must be identical. The resident never types a
`site_id` or a URL; they pick a provider from the dropdown and enter only
username and password.

## `run_scrape(creds)` contract

- `creds` is `{ username, password }`, read from the device keystore at scrape
  time. The login URL comes from the `// @url` header, not `creds`.
- Return shape:
  - `bills`: array of `{ due_date (ISO YYYY-MM-DD), total (number) }`
  - `usage`: array of `{ account_ref, period_start, period_end, amount, unit }`
  - Return empty arrays for data a site does not expose.
- Throw (or reject) to surface an error to the host.

## How the generated script runs

The WebView re-injects the script on every page load. State is carried across
injections in `window.name` (survives cross-origin navigation; empty in a fresh
WebView, so each job starts at step 0).

- `NAV_STEPS` is the recorded login flow. Step types: `goto`, `fill_credential`
  (`which: "username" | "password"`), `fill_text`, `wait_for`, `click`. Each
  element is located by the recorder's captured identity (id, then name,
  aria-label, placeholder, then role + visible text), not a brittle CSS path.
- `run_scrape` reads the step pointer from `window.name`, runs steps in the
  current load until one navigates, advances the pointer, and returns a
  never-settling promise so no result posts until the next injection resumes at
  the advanced pointer.
- When the pointer passes the end of `NAV_STEPS`, `run_scrape` clears
  `window.name` and calls `EXTRACTION`, retrying while JS-populated billing
  fields render.

## Host-provided helper

- `window.__set_value(el, value)` sets an input's value without focusing it, so
  the keyboard does not pop on the hidden WebView. Generated scripts use it
  instead of assigning `el.value` or calling `el.focus()`.
